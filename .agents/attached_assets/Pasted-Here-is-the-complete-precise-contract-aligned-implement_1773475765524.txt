Duration: 2 weeks — the most technically careful phase

4A — ProjectSerializer (new Side System, read-only)
New file: src/core/persistence/ProjectSerializer.ts

Classification: Side System (alongside BimManager, SemanticGraph). Contract compliance:

Reads stores via store.getAll() — read-only, never writes
Never calls builders
Never fires Store Event Bus
Is not part of the command/store/builder flow
Serializes all stores to a plain JSON snapshot:

ProjectSnapshot {
  version: number -- schema version for migration
  savedAt: number -- timestamp
  savedBy: string -- user ID
  projectId: string
  levels: LevelSnapshot[] -- BimManager level data
  stores: {
    walls: WallData[] -- serialized, THREE.js stripped to {x,y,z}
    slabs: SlabData[]
    doors: DoorData[]
    windows: WindowData[]
    columns: ColumnData[]
    beams: BeamData[]
    stairs: StairData[]
    curtainWalls: CurtainWallData[]
    furniture: FurnitureData[]
    plumbing: PlumbingData[]
    roofs: RoofData[]
    handrails: HandrailData[]
    grids: GridData[]
  }
  semanticGraph: GraphSnapshot -- all relationships
  bimManager: BimManagerSnapshot -- levels, elevations
}
THREE.js serialisation rule: Every THREE.Vector3 becomes {x,y,z}. Every THREE.Euler becomes {x,y,z,order}. No THREE.js class instances in the JSON. This must be explicit for every store type — tested individually.

4B — ProjectLoader (new Side System)
New file: src/core/persistence/ProjectLoader.ts

Loads a ProjectSnapshot by dispatching existing Create* commands through CommandManager. No direct store mutation — contract mandatory.

Load sequence (order matters — respects contract and element dependencies):

1. ClearProjectCommand (new — clears all stores, resets scene)
2. AddLevelCommand × N (levels first — everything references levelId)
3. AddGridCommand × N
4. CreateColumnCommand × N
5. CreateWallCommand × N
6. CreateSlabCommand × N
7. CreateBeamCommand × N
8. CreateStairCommand × N
9. CreateRoofCommand × N
10. CreateCurtainWallCommand × N
11. CreateWindowCommand × N (windows reference walls)
12. CreateDoorCommand × N (doors reference walls)
13. CreateFurnitureCommand × N
14. CreatePlumbingFixtureCommand × N
15. CreateHandrailCommand × N
16. RestoreSemanticGraphCommand (new — restores relationships after all elements exist)
Each Create* command flows through:
CommandManager.execute() → Store.set() → Store Event Bus → DependencyResolver → Builders → Scene

This is identical to a user creating every element manually. The scene rebuilds deterministically. The contract is fully preserved.

6A — WebSocket Server
New server file: server/realtime/SocketServer.ts

Uses Socket.io. Rooms are keyed by project_id.

Events — Client → Server:

join_project {projectId, userId, sessionId} → validate permission → join room → broadcast presence
leave_project {projectId, sessionId} → leave room → broadcast departure
execute_command {projectId, command: SerializedCommand, clientTs} → validate → store in DB → broadcast to room
cursor_move {projectId, position: {x,y,z}} → broadcast to room (not stored)
ping → update last_ping_at
Events — Server → Client:

user_joined {userId, displayName, avatarUrl, sessionId} → show avatar in presence bar
user_left {sessionId} → remove avatar
command_broadcast {command: SerializedCommand, userId, serverSeq} → client replays
cursor_update {sessionId, userId, position} → update cursor indicator
error {code, message} → show notification
6B — Client-Side Command Sync
New file: src/core/realtime/RealtimeCommandSync.ts

Classification: UI/Tool layer (dispatches to CommandManager, never to stores directly). Contract compliance: all incoming remote commands go through CommandManager.execute(cmd, {source: 'REMOTE_USER', userId}).

Local execute flow:

User action → CommandManager.execute(cmd) → scene updates locally
            → cmd.serialize() → WebSocket → server → broadcast
Remote receive flow:

WebSocket message → deserialize to Command instance (CommandType → class map)
                 → CommandManager.execute(cmd, {source: 'REMOTE_USER'})
                 → Store → Store Event Bus → Builders → Scene
This is fully contract-compliant. The remote command flows through the identical path as a local command.

Conflict handling:

Different elements: no conflict, both apply
Same element, last-write-wins: the server server_seq defines authoritative order
Deleted element edited remotely: canExecute() fails, show notification to remote user

The Three Things That Require Your Oversight
Every other phase can be implemented by an AI agent with precise prompts. These three require your architectural review:

1. ProjectSerializer — Phase 4A.
All 13 store types need exact serialisation. WardrobeConfig nested structures, THREE.js Vector3/Euler objects, wall curve descriptors, semantic graph relationships. One wrong field means a corrupted save file. Test each store type independently before moving to the next.

2. ProjectLoader load sequence — Phase 4B.
The order in which elements are loaded matters — doors reference walls, walls reference levels. The PlanOrdering.ts file in your codebase already defines command type ordering — that is the authoritative input to the load sequencer. Verify it covers all 13 element types.

3. Remote command deserialization — Phase 6B.
The map from CommandType enum value back to the concrete command class is the critical piece of multi-user sync. Every command type needs a registered deserializer. This must be complete — a missing entry means that command type is silently dropped in multi-user sessions, corrupting the shared state.
