PRYZM Monetization — Full Implementation Blueprint
This plan maps directly to your existing architecture. Every section names the exact files to create or modify and explains why, so there is no ambiguity when coding begins.

Architecture Overview
The monetization system has four independent layers that must be built in order, because each one depends on the previous:

Layer 1: Identity & Subscription (Who is this user? What plan are they on?)
    ↓
Layer 2: Entitlement Store (What is this user allowed to do right now?)
    ↓
Layer 3: Enforcement Gates (Intercept forbidden actions before they execute)
    ↓
Layer 4: UI — Prompts, Paywalls, Counters (Show the user what's happening)
Layer 1 — Identity & Subscription
1.1 Database Schema (Supabase)
You already have Supabase wired in server.js. You need three new tables alongside your existing projects and project_versions.

On the frontend, you need to add the Clerk frontend SDK to index.html or main.ts. Clerk provides a JS snippet that attaches to the DOM and manages session tokens. Every API call from the frontend should include the Clerk session token in the Authorization: Bearer <token> header.

1.3 New Server Routes
New route: GET /api/me
Called on every app load. Returns the full user context:

{
  "user": { "id": "...", "email": "...", "displayName": "..." },
  "plan": "architect",
  "planStatus": "active",
  "trialEndsAt": null,
  "workspace": { "id": "...", "name": "...", "role": "admin", "members": [...] },
  "aiUsage": { "used": 23, "limit": 50, "period": "2026-03" },
  "aiCredits": { "remaining": 100 }
}
New route: GET /api/ai/usage
Returns AI consumption for the current billing period. Reads from ai_usage table filtered by user_id and period_start = first day of current month.

New route: POST /api/ai/log
Called by the AI proxy after every Claude call. Logs the usage to ai_usage table. This is where you track tokens and compute cost.

New route: POST /api/stripe/checkout
Creates a Stripe Checkout session for a given plan. Returns a checkoutUrl that the frontend redirects to.

New route: POST /api/stripe/webhook
Stripe webhook endpoint. Handles checkout.session.completed, customer.subscription.updated, customer.subscription.deleted, and invoice.payment_failed. Updates the users table plan and status accordingly. This is the source of truth for subscription state.

New route: POST /api/workspaces
Creates a new workspace. Only callable by Studio/Firm/Enterprise users. Sets owner_id to current user and initializes seat_limit based on plan.

New route: POST /api/workspaces/:id/invite
Sends an email invite to join a workspace. Creates a pending workspace_members row.

New route: DELETE /api/workspaces/:id/members/:userId
Removes a member. Only workspace owner or admin can do this.

Modified route: POST /api/anthropic/v1/messages
This route currently just proxies to Anthropic with no tracking. It must now:

Validate the user is authenticated (reject anonymous)
Check AI usage against the plan limit via checkAIQuota(userId) — a utility function reading from Supabase
If over quota, check ai_credits table for remaining purchased credits
If both exhausted, return HTTP 402 with { error: 'ai_quota_exceeded', upgradeUrl: '/pricing' }
If allowed, make the Anthropic request
After response, call POST /api/ai/log with token counts from the response's usage field
Return the response to the frontend
Important: Your AIElementFactory.ts currently calls a Cloudflare Worker directly, bypassing your server entirely. Both factories must be redirected to go through /api/anthropic/v1/messages instead. This is how you gain enforcement control.

Layer 2 — Entitlement Store
New file: src/auth/EntitlementStore.ts

This is the single source of truth for what a user can do. It reads from AuthStore and exposes pure computed properties. No gate should ever compute entitlements itself — all checks go through here.

// Computed from AuthStore.plan + AuthStore.planStatus
const EntitlementStore = {
  // Projects
  maxProjects: computed,          // 3 (free) | Infinity (paid)
  canCreateProject: computed,     // false if at limit
  // Export
  canExportIFC: computed,         // false on free
  canExportGLB: computed,         // false on free
  // AI
  aiActionsLimit: computed,       // 5 | 50 | 200 | 500 | Infinity
  aiActionsUsed: computed,        // from live usage tracking
  aiActionsRemaining: computed,   // limit - used + credits
  canUseAI: computed,             // remaining > 0
  canUseDesignAdvisor: computed,  // free gets limited version
  canUseFloorPlanAI: computed,    // paid only
  canUseElementCreator: computed, // paid only
  canUseWardrobeFactory: computed,// paid only
  // Geospatial
  canUseCesium: computed,         // paid only
  // Collaboration
  canInviteMembers: computed,     // Studio+ only
  maxSeats: computed,             // 1 | 1 | 8 | 25 | -1
  currentSeats: computed,         // from workspace members
  // Version History
  maxVersions: computed,          // 0 (free) | 15 | Infinity | Infinity
  // Trial
  isOnTrial: computed,            // planStatus === 'trialing'
  trialDaysLeft: computed,
  // Upgrade
  needsUpgrade(feature: Feature): boolean,
  suggestedPlan(feature: Feature): Plan,
}
The Feature enum covers every gated capability:

enum Feature {
  IFC_EXPORT, GLB_EXPORT, AI_DESIGN_ADVISOR, AI_FLOOR_PLAN,
  AI_ELEMENT_CREATOR, AI_WARDROBE, CESIUM_GIS, COLLABORATION,

Layer 3 — Enforcement Gates
Gates intercept actions before execution and either allow them, block them with an upgrade prompt, or degrade gracefully.

3.1 Command-Level Gate
Modified file: src/commands/CommandManager.ts

Before commandManager.execute(command) runs, add a pre-execution hook:

CommandManager.execute(command):
  1. Look up command.type in FeatureRequirements map
  2. If command requires a feature, call EntitlementStore.needsUpgrade(feature)
  3. If blocked: dispatch 'feature-blocked' event with { feature, command }
     → Do NOT execute the command
     → Do NOT add to history
     → Return { blocked: true, reason: Feature.X }
  4. If allowed: proceed with existing execute() logic
New file: src/auth/FeatureRequirements.ts
A map from command type to required feature:

const FeatureRequirements: Record<CommandType, Feature | null> = {
  ExportIFCCommand:        Feature.IFC_EXPORT,
  ExportGLBCommand:        Feature.GLB_EXPORT,
  InitCesiumCommand:       Feature.CESIUM_GIS,
  SaveVersionCommand:      Feature.VERSION_HISTORY,
  CreateProjectCommand:    Feature.UNLIMITED_PROJECTS, // checked against limit
  // null means no restriction
  CreateWallCommand:       null,
  CreateSlabCommand:       null,
  // ... all other commands
}

5.2 Socket.io Multi-User State
Modified file: server.js (Socket.io section)

The existing join-project / command-executed / cursor-move structure is already correct. Additions needed:

On join-project: server validates workspace membership, then emits room-state back with the list of currently-connected users in that project room
On command-executed: server validates the emitting socket has at least member role before rebroadcasting (prevents rogue clients)
Add user-presence event: periodically (every 30s) each client emits its cursor position and active tool; server aggregates and broadcasts to room
New event: seat-limit-reached
When a workspace's connected user count exceeds seat_limit, new connection attempts receive this event and are not admitted to the room.

5.3 Conflict Resolution (Concurrent Edits)
Your command-executed Socket.io event already broadcasts BIM changes to collaborators. The challenge is concurrent edits (two users modify the same wall simultaneously). The recommended approach for your architecture:

Each CommandProposal from a remote user arrives as a command-executed socket event. The receiving client:

Checks if it has any pending local commands that touch the same element ID
If conflict: the server-side user (whoever emitted command-executed first) wins; the local client queues its command as a pending retry
The CommandManager already has transactionality via snapshots — use this to safely apply remote commands without corrupting local state
This is "last-write-wins at the element level," which is appropriate for BIM (unlike Google Docs where character-level merging matters).

6.2 Checkout Flow
User clicks "Upgrade to Architect"
  → Frontend calls POST /api/stripe/checkout { plan: 'architect', interval: 'monthly' }
  → Server creates Stripe Checkout Session:
      customer: stripe_customer_id (from users table, or create new)
      line_items: [{ price: STRIPE_PRICE_ARCHITECT_MONTHLY, quantity: 1 }]
      success_url: https://app.pryzm.io/app?checkout=success
      cancel_url: https://app.pryzm.io/pricing
      metadata: { userId: req.auth.userId, plan: 'architect' }
  → Server returns { checkoutUrl }
  → Frontend: window.location.href = checkoutUrl
  → Stripe handles payment
  → Stripe fires webhook to POST /api/stripe/webhook
  → Webhook handler updates users.plan = 'architect', users.plan_status = 'active'
  → User returns to app, AuthStore re-fetches /api/me, plan updated
6.3 Webhook Handler (Critical Path)
The webhook is the only reliable way to know a payment succeeded. It must handle:

checkout.session.completed → set plan active, create workspace if Studio/Firm
customer.subscription.updated → sync plan changes (upgrades, downgrades)
customer.subscription.deleted → downgrade to free, mark workspaces as suspended
invoice.payment_failed → set plan_status = 'past_due', email user (via Clerk or SendGrid)
invoice.payment_succeeded → confirm active, reset plan_status if was past_due
Always verify the Stripe webhook signature using your STRIPE_WEBHOOK_SECRET before processing.
