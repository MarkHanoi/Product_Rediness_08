Wall System Classification: B — Mostly Command-First
Justification:
The wall system is overwhelmingly command-first at the mutation layer. The WallTool never touches the store directly, never registers spatial elements, and always delegates creation to CreateWallCommand. However, the builder (WallFragmentBuilder.buildWall()) resolves world position via spatialAuthority.resolveWorldTransform() which internally reads all semantic stores through (window as any).wallStore — a cross-layer coupling that partially undermines §4 (Projection Layer must not access semantic layer directly). Additionally, active (window as any) fallbacks in WallTool and SpatialAuthority lower the classification below Fully Command-First.

There are therefore two separate elevation computation paths — the correct one in updateWall() (direct BimManager lookup) and the actual one used for positioning in buildWall() (via SpatialAuthority with window-global fallback). If these diverge (e.g., wall not yet registered), the worldY log in updateWall() would mislead debugging while the actual geometry is misplaced.

✅ Scene is searched for orphan roots before creating new ones.
✅ Children cleared before rebuild via wallGroup.clear().

✅ id, elementType, levelId, baseLine, height, thickness all set early in buildWall() before any early-return branch.
✅ Fields are refreshed from semantic state on every rebuild.

🟡 Fragment IDs are generated via crypto.randomUUID() inside buildWall() on every call. 1️⃣2️⃣ Risk Summary
🔴 Critical Architectural Violations
CreateWallBetweenMarksCommand — Non-deterministic redo (§2.4)
execute() computes wall baseline by taking midpoints of the CURRENT positions of source walls. On redo, if source walls have been modified, the midpoints will differ, producing a geometrically different wall than the original execution. Redo is not idempotent.

SpatialAuthority.getSemanticData() reads from (window as any).wallStore (§4, §12)
WallFragmentBuilder.buildWall() → spatialAuthority.resolveWorldTransform() → getSemanticData() reads semantic state from window globals. This is a Builder → Semantic Layer cross-layer violation. The builder must remain a pure projection function with no semantic data access.

SpatialAuthority.resolveWorldTransform() fallback to L0 elevation (§13)
When an element is not found in any BIM level, the authority falls back to getLevelById('L0') || levels[0] and continues with the wrong elevation. This is a forbidden elevation fallback per §13.

SpatialAuthority reconciliation listener — Direct scene mutation bypassing pipeline (§2.1, §4)
On level elevation change, the reconciliation listener directly mutates obj.position on scene objects without going through the store → event bus → builder path. This is an architectural bypass.

Split-brain worldY in WallFragmentBuilder (§13)
updateWall() correctly computes worldY = level.elevation + baseOffset and logs it but never passes it to buildWall(). buildWall() independently derives Y from spatialAuthority.resolveWorldTransform(). Two separate computation paths exist; if they diverge the logged value is misleading while the actual geometry is wrong.

🟠 Major Design Weaknesses
UpdateWallDimensionsCommand — structuredClone on THREE.Vector3-containing frozen objects (§2.2)
structuredClone called on objects containing class instances is fragile. Environments that throw on non-serializable objects would crash execute() before the baseLine rescue. The serializeWallSnapshot() pattern used by other commands is the correct approach.

Eight active window-global fallbacks in WallTool (§1.1)
bimManager and commandManager are still resolved via (window as any) at 8 call sites if not injected. Migration is partially complete but callers that haven't been updated silently degrade rather than fail fast.

createFromSelectedSlab() — selectionManager and slabTool accessed via window globals (§1.1, §23.1)
These two cross-tool dependencies are not in WallToolCallbacks and have no injection path.

WallFragmentBuilder.fragmentToEntityMap not purged on rebuild (memory leak)
removeWallFragments() clears wallToFragmentsMap and fragments but fragmentToEntityMap entries are only added, never removed on rebuild. Stale entries accumulate indefinitely.

Hosted element registry orphan on wall delete
CreateWallCommand.undo() unregisters only the wall from bimManager and elementRegistry. Hosted opening elements (windows/doors) registered via CreateWallOpeningCommand are never unregistered from bimManager when their parent wall is deleted, creating orphaned spatial registrations.

🟡 Minor Improvements
WallStore.update() stamps new modifiedAt on undo restore paths
Every undo call via wallStore.updateWall() advances modifiedAt and increments version, corrupting the audit trail. Undo should ideally restore the original metadata snapshot.

UpdateWallVisualPropertiesCommand — Global scope (all walls)
Targets all walls in the store. There is no scoped filter (by level or selection). Intentional but architecturally risky for large models.

CreateWallOpeningCommand — Unused prevWallSnapshot field
Captured in execute() but never used in undo() (which uses removeOpening() instead). Dead code that adds maintenance surface.

UpdateWallHeightCommand — CustomEvent via window.dispatchEvent (§12)
emitWallUpdatedEvent() and triggerAIRefresh() use raw window events as a side-effect channel outside the documented Store Event Bus, creating an undocumented coupling to AI and inspector listeners.

WallStore.getLevelById() hardcodes height: 3.0
The BimKernel's actual level height is discarded; any consumer of WallStore.getLevelById() who reads .height gets a constant 3.0 regardless of the real level height.

WallToolCallbacks.bimManager and commandManager are typed any
Should be typed to their concrete interfaces to enforce the injection contract at compile time.

CreateWallCommand spatial ordering — store.add → bimManager.register → elementRegistry.register. Correct per §5.
