## 0. The mental model in one paragraph

A house is generated by **reusing the FROZEN apartment layout engine (D-TGL) once per storey**,
wrapped by a **house orchestrator** that splits the brief across storeys, reserves a vertical stair
core, and stitches the per-storey results into a building (perimeter, slabs, stair, voids, roof).
Each storey is laid out **deterministically** (no RNG): the engine **enumerates 8 fixed symmetry
strategies**, subdivides the plate into room rectangles, scores each candidate against ~21 weighted
objectives, and picks the Pareto-best. Then it **emits walls, doors and windows**, which the
**editor executor** dispatches as commands; the editor's **WallJoinResolver** mitres the walls and
**RoomDetectionEngine** re-derives the rooms from the closed wall loops. Almost every current defect
is a break in ONE of these hand-offs.

## 2. The data model & relationships (what flows through the spine)

```
Brief ──► Program ──► [per storey] ──► BubbleGraph ──► Rects ──► RoomPlacements ──► Openings ──► Walls
 (sliders)  (room set)   (allocate +     (targets +    (decompose   (squarify /        (doors +     (editor
            §G13 not       enrich)        adjacency +    − stair)     carve, drop      windows)     mitre +
            wired)                        minima)                     report)                       detect)
                                                                                                       │
                                                            Rooms (detected) ◄────── closed loops ◄────┘
```

**Key relationships (and the gap each break causes):**
- **Program → Zones:** privacy gradient (public→circulation→private→service). *Break → bedroom in
  the social zone (G2/G3).*
- **BubbleGraph targets ↔ Plate area:** room sizes scaled to the AVAILABLE area. *Break → §AREA-
  AGREEMENT starve → drops (G12).*
- **Rects ↔ Stair:** the stair keep-out subtracts from the plate. *Break → fragmentation → cram →
  drops (G12) + stair pokes out (G8).*
- **RoomPlacement → Frontage:** `frontage:'required'` rooms must touch the shell. *Break → interior
  habitable room → NO window (the missing-windows defect).*
- **Walls → WallJoinResolver:** the resolver trims/extends endpoints. *Break → wall not closing →
  rooms merge (G1) or door overruns a trimmed wall → sealed (was the bathroom defect).*
- **Closed loops → RoomDetection:** rooms are re-derived from walls, NOT from the engine's
  placements. *Break → a dropped-room void is detected as a generic "Room 00-00x" (G9).*

> **Critical subtlety:** the engine's room placements are NOT the final rooms. The editor RE-DETECTS
> rooms from the built walls. So a perfect engine plan can still produce wrong rooms if the walls
> don't close exactly where the engine put them. This is why wall-closing (G1) is the keystone.

## 3. The engine (D-TGL): how a layout is chosen

`generateDeterministicLayouts` → `enumerateLayouts`. **No RNG, no optimiser loop** — a fixed
enumeration ranked by Pareto dominance + weighted sum.

### 3.1 The 8 strategies (`enumerate.ts` `STRATEGIES`)
`{axis: swap x↔z, order: fwd|rev, mirror: yes|no}` = 8 fixed symmetry transforms. (+ a 9th
double-loaded parti exists behind A.27.) Each physically relocates rooms to different façades, so
**orientation diversity already exists across the 8 candidates** — the ranking picks among them.

### 3.2 Decompose + stair-subtract (7c)
`decomposeToRects(polygon)` → axis-aligned rects (a rotated rectangle → 1 rect in the principal-axis
frame). `subtractRectsFromRects(rects, stairKeepOut)` → **the stair fractures the plate**; sets
`stairCarved`. **This is the G12 fragmentation source.**

### 3.3 Subdivide (7d) — `subdivideWithReport`
Decision tree (the branch is logged by `§DIAG-RECTS`/`§DIAG-BRANCH`):
1. `stairCarved && valid≥2` → if a rect holds ≥ `DOMINANT_FRACTION` (now **0.45**, was 0.55) of the
   area → **carve a corridor spine in the dominant rect** (`trySingleRectCarve`) AND run
   `packMultiRect`, keep whichever drops fewer (`§STAIR-CARVE-NO-DROP`).
2. single rect → `trySingleRectCarve` (public | corridor | private) or double-loaded.
3. else → `packMultiRect`: rooms allocated to rects ∝ area; **last rect absorbs the remainder** →
   if a fragment is too small for its slice, `§FEASIBILITY-ALLOC` drops the lowest-priority room.

Inside a rect: `tryCarveCorridor` (public/private bands + corridor strip), `squarify` (treemap to
area targets), `applyMasterSurplus`, `tryCarveEnsuiteFromMaster`, `reshapeCorridorStrip`
(§CORRIDOR-PHYSIOGNOMY: narrow a fat corridor to a strip).

### 3.4 §FEASIBILITY-ALLOC (the drop) — `placeInRectReported`
A rect can't hold all its rooms' **minimums** (`minAreaFor` from `programRules`), OR a room fails its
**short-side floor** after squarify → drop the lowest **drop-rank** room (`dropRankFor`: service
before habitable; never a bathroom/bedroom before a lower-priority service room). Reported in
`droppedRooms` (never silent). **The misleading log** ("rect 51 < Σ 14 for 1 room") prints the
original rect area while the real failure is a sliver short-side, not area (see G12 audit).

### 3.5 Objectives + Pareto (7e) — `objectives.ts` + `envDrivers.ts`
~21 axes, each weighted by a priority band: efficiency, adjacency (`preferenceBetween`), frontage
(`validateFrontage` — HARD reject if a `frontage:'required'` room has no shell wall), **solar
orientation** (`solarOrientationScore` — daytime rooms to the equator side; **bedrooms NOT scored
today**), **acoustic zoning** (soft), circulation (`circulationRouted` HARD — every private room
reaches the spine; sequence quality soft), shape/aspect, topology. **Pareto-non-dominated set →
weighted sum → winner.** Ties broken by fixed strategy order (determinism).

## 4. The house orchestration (the wrapper around D-TGL)

`houseOrchestrator.ts` — the storey loop:
1. `allocateProgramToStoreys(brief, storeyCount)` — bedrooms/baths upstairs, social downstairs.
2. `reserveStairCoreShaped(footprint, risers)` — ONE vertical core, same rect every storey
   (alignment invariant). Shape I/L/U by available box; position by `chooseStairCorePosition`
   (§STAIR-WORST-ASPECT: hug the poor-aspect perimeter wall) + `snapRectInsidePoly` (keep inside a
   rotated shell). **§STAIR-RUN-BOUND** sizes the I-run depth to risers×tread (the L/U overrun G8
   remains).
3. Per storey: `enrichStoreyProgramToPlate` (raise a sparse program to fill the plate:
   `growBedrooms` upper, `growGroundRooms` multi-storey ground, **`entranceHall:true` on upper →
   the G14 "Entrance Hall on floor 1" bug**), `houseStoreyBand` → `presentedAreaM2`
   (**§AREA-AGREEMENT**: don't cap below the true plate for an enriched storey), then
   `generateDeterministicLayouts`.
4. Stitch: ground reuses the pre-drawn shell + `_weldGroundPartitions` (welds interior endpoints to
   the shell — **ground-only**, why ground walls are good and upper aren't); upper storeys get
   `_buildPerimeterShell` (fresh perimeter, shares endpoints with partitions). Then stair, stairwell
   voids (`§VOID-FINISH`), roof, per-storey floor-plan views (`§FLR-VIEWS`).

## 5. Openings — doors & windows

### 5.1 Doors (`wallsAndDoors.ts`)
`repairSegments` first (weld coincident endpoints; **§SELF-CLUSTER-FLOOR** drops <0.50 m stubs the
resolver would self-cluster). Then door passes, in order: (1) bubble-requested legal doors,
(2) primary-access reconcile, (3) permitted reconcile, (4) over-cap last resort,
(5) **§CIRCULATION-REROUTE** multi-hop BFS so every room is reachable. `addDoor` rejects a wall
shorter than the room-type door floor (`minDoorWidthBetween`); `findClearOffset` slides the door off
junctions (§DOOR-APPROACH-QUALITY). **Entrance door:** `entranceDoor.ts` `resolveEntranceDoor`
(hall-fronting shell wall, **§ENTRANCE-DOOR-CLEAR** gap-aware to dodge windows). **§DOOR-LIVE-CLAMP**
(editor) re-clamps every door to the LIVE post-trim wall so a resolver trim can't seal a room.

### 5.2 Windows (`windowEmission/*`)
**Per-room, longest-external-wall placement** (`emitWindowsForRoom`): each room emits up to
`MAX_WINDOWS_PER_ROOM` on its OWN external walls, ranked by `length × solarMultiplier(1.0–1.6)`.
`resolveAllShellWindows` + `deOverlapShellWindows` host them on the shell. **The two structural
limits:** (a) a room with NO external wall gets NO window → the missing-windows defect couples to
frontage (G11/W4); (b) length dominates orientation + no cross-room façade balancing → windows
cluster on the longest façade(s). `windowMandatory` (living/kitchen/master/bedroom = true;
dining/study/wet = false) is a HARD reject axis — but only fires if the room IS on the perimeter.

## 6. Invariants (the relationships that MUST hold)

| Invariant | Where enforced | Break = gap |
|---|---|---|
| **Every-room-access** — every room reachable by a door | `§SEALED-ROOMS` + `§CIRCULATION-REROUTE` + `§DOOR-LIVE-CLAMP` | sealed bathroom (fixed v68) |
| **Open-plan-eligible** — only living/kitchen/dining may merge wall-less | `isOpenPlanEligible` | "Living/Bedroom" blob |
| **Privacy gradient** — private deeper than social | zoning + `validateFrontage` (no orientation) | bedroom at the entrance (G2/G3) |
| **Frontage-required** — living/kitchen/master/bedroom touch the shell | `validateFrontage` HARD reject | interior room → no window |
| **Window-mandatory** — living/kitchen/master/bedroom have a window | objective HARD reject | dark room |
| **Door caps** — bedroom 1, master 2, wet 1 | `maxDoorsFor` | through-circulation |
| **Determinism** (ADR-0061) — same input → byte-identical output | no RNG, fixed strategy order, gated-when-absent | non-reproducible |
| **Stair alignment** — same core rect every storey | `reserveStairCore` (footprint-only) | mis-stacked stair |

## 8. The diagnostic surface (what each §DIAG tells you)

Paste the console; grep these:
- `§DIAG-STOREY` — true vs presented area, program set, stair shape/pos → **is the plate starved or the stair central?**
- `§DIAG-ALLOC` / `§DIAG-ENRICH` — brief → per-storey split → what enrich added → **is the wrong room set on this floor?**
- `§DIAG-BUBBLE` — per-room target/min/privacy → **are targets sized to the plate?**
- `§DIAG-RECTS` / `§DIAG-BRANCH` — fragments, dominant fraction, carve-vs-generic, dropped types → **is fragmentation dropping rooms?**
- `§DIAG-ENUM` / `§DIAG-WINNER` — per-candidate drops/scores, winner per-axis breakdown, frontage-fail rooms → **why this layout won; which rooms have no shell wall.**
- `§DIAG-WIN` / `§DIAG-WIN-DIST` — per-room windows + why-zero + façade distribution → **which rooms are unglazed and why.**
- `§DIAG-DOORS` — per-door pass, sealed/rerouted list → **accessibility.**
- `§DIAG-STAIR` — candidate scoring, corner vs mid-edge → **fragmentation predictor.**
- `§DIAG-ROOMS` (editor) — per-room type/area/windowCount + ⚠ WINDOWLESS list → **the final built truth.**

## 9. Target architecture (v2.0) vs current — the roadmap, not more fires

The v2.0 spec wants a **constructive** pipeline; today several steps are **scoring-only** (they rank
but don't place). The strategic order to close the gap:

1. **Stop dropping rooms (G12).** Cure the stair fragmentation (place the stair flush in a corner so
   the plate keeps a dominant rect) + per-fragment program sizing. *Everything else needs real rooms.*
2. **Close upper-floor walls (G1).** Extend `_weldPartitionsToShell`-equivalent to `_buildPerimeterShell`.
3. **Constructive frontage allocation (W4 → windows everywhere).** Swap interior `frontage:'required'`
   rooms to the perimeter so every habitable room can be glazed; then per-façade window distribution.
4. **Sun-aware placement (§10).** Score bedrooms east; bias living/kitchen to the sunny façade.
5. **Walking-distance proximity (§9) + acoustic placement (§4).** Turn the soft axes into real
   adjacency/path/sound coherence.
6. **Wire the brief (G13).** Target-area, style, master-floor into generation.

Each step is a **stage** in §1, not a patch. Implement against this MASTER + the v2.0 rules, verify
with the §DIAG surface, keep ai-host green. That is the vision: an orchestrated algorithm, not a
sequence of fire-fights.
