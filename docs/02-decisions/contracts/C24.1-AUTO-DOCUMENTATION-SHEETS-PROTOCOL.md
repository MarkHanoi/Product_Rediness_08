# C24.1 — Auto-Documentation Sheets Protocol

> **Stamp**: 2026-06-09 · **Status**: DRAFT
> **Scope**: governs the AUTOMATIC generation (vs interactive authoring) of a documentation
> SET for a generated/authored building — per-level plans, building elevations, per-room
> cropped plans + interior elevations, and set-out plans — and their auto-placement on
> numbered sheets ready for PDF. Covers the deterministic decision layer (`DOC-AUTO`
> `DS1`–`DS6`) + the editor executor that realises it. It does NOT re-define the sheet,
> view, dimension, or PDF primitives — those are governed by the contracts below; this
> contract governs how they are AUTO-DRIVEN.
> **Depends on**: [C24](C24-SHEET-COMPOSITION-ENGINE.md) (sheets/viewports/title blocks),
> [C29](C29-PDF-VECTOR-EXPORT.md) (vector PDF), [C30](C30-DRAWING-SET-MANAGEMENT.md)
> (drawing sets / numbering / revisions), [C34](C34-PRINT-AND-DRAWING-STANDARDS.md)
> (sizes/scales/line-weights/dimension styles), [C16](C16-COMMAND-AUTHORING-PROTOCOL.md).
> **Key principles**: **P5** (the DS1–DS6 decision layer is PURE — no I/O/DOM/THREE/RNG),
> **P6** (the executor mutates ONLY via the command bus), **P8** (a span at the workflow
> boundary + per C29 a span per PDF export), **ADR-0061** (deterministic: same input →
> byte-identical plan).
> **Plan**: [AUTO-DOCUMENTATION-SHEETS-PLAN.md](../03-execution/plans/AUTO-DOCUMENTATION-SHEETS-PLAN.md) + master-tracker §24.

## §1 — Invariants

### §1.1 — The decision layer is PURE and deterministic
The "what documents to make" decision is a pure function chain (`DS1` `buildSheetFromViews`,
`DS3` `computeBuildingElevationMarks`, `DS4` `roomCropRegion` / `computeRoomInteriorElevationMarks`,
`DS5` the `set-out` dimension producer mode, `DS6` `planDocumentationSet`). These live in pure L2/L4
packages (`@pryzm/ai-host`, `@pryzm/geometry-kernel`, `@pryzm/drawing-primitives`), import no
store/DOM/THREE, use no `Math.random`/`Date.now`, and return byte-identical output for the same
input. Timestamps/revision stamps are applied by the executor AFTER the plan returns, never inside
the pure core (ADR-0061).

### §1.2 — The executor is the ONLY mutation path (P6)
The editor `docSheets` executor realises `planDocumentationSet`'s output by DISPATCHING commands —
`view.createDefinition` (one per `DocViewSpec`), the sheet handlers (`sheet.create` / `sheet.addViewport`
/ `sheet.setTitleBlock`), the drawing-set handlers, and `sheet.export.pdf`. It MUST NOT write any
store directly. One run = one logical unit (a `batchCoordinator.runBatch` where applicable) so undo
removes the generated set coherently.

### §1.3 — Coverage rule (per-level + per-room)
A documentation set MUST cover: (a) one PLAN sheet per level; (b) one BUILDING-ELEVATIONS sheet
(the 4 N/S/E/W exterior elevations); (c) one SET-OUT plan per level; (d) one ROOM sheet per
DETECTED room (a cropped plan + up to 4 interior elevations). A room that is degenerate (no valid
crop region) is SKIPPED, and the skip MUST be logged (`§DIAG` + the set's coverage report) — silent
omission is forbidden (a missing room sheet must be explainable).

### §1.4 — Numbering scheme
Sheets are numbered within a single architectural discipline `A`, by phase:
`A-1xx` level plans · `A-2xx` building elevations · `A-3xx` set-out plans · `A-4xx` room sheets.
Numbering is deterministic (levels in elevation/input order; rooms grouped by level then id) and
sequential within each phase (`A-101`, `A-102`, …). Re-running on an unchanged model yields the
SAME numbers. (Numbering authority is C30; this contract fixes the AUTO scheme that feeds it.)

### §1.5 — Label/dimension placement is RULE-BASED (v1)
Auto-placement of dimensions, room labels, area tags, and elevation marks uses DETERMINISTIC
RULE-BASED placement (e.g. `set-out` dims keyed to a wall start; interior elevation marks at the
room centroid; building elevation marks offset outside each façade). Force-directed / collision-
avoiding label placement is explicitly OUT OF SCOPE for v1 (deferred per SPEC-04 §8.2). A v1 set may
overlap a label in a dense plan; that is acceptable and MUST NOT be "fixed" by adding RNG.

### §1.6 — Standards + style compliance (C34)
Every emitted sheet/dimension/elevation honours the active `StandardsStore` (C34): paper sizes,
scales, line weights, dimension + text styles, north arrow, scale bar. The auto layer chooses
WHICH views/dims to emit; their STYLE is C34's. No hard-coded sizes/weights in the DS layer.

### §1.7 — PDF is vector-only + spanned (C29)
Export goes through the C29 vector PDF backend (`sheetToPdfBytes` / the book exporter). No raster
fallback for linework; every export emits an OTel span (P8). A multi-sheet set exports as a book
(one PDF, sheets in numbering order).

### §1.8 — Idempotent re-generation
Re-running auto-documentation on a model that already has an auto-set MUST dedupe (reuse an existing
plan ViewDefinition for a level — as §FLR-VIEWS already does — and not duplicate sheets/numbers).
The executor checks for an existing auto-set before creating, and updates rather than duplicates.

## §2 — Surfaces (AS-IS → the auto layer)

| Concern | Existing (governed by) | Auto layer (this contract) |
|---|---|---|
| Sheet / viewport / title block | C24 (`plugins/sheets`, `drawing-primitives`) | `DS1 buildSheetFromViews` chooses scale + lays viewports out |
| Per-level plan view | §FLR-VIEWS (HouseLayoutExecutor) | reused; `DS6` emits one plan sheet per level |
| Building elevation projection | EdgeProjectorService + CreateElevationMarkCommand | `DS3` computes the 4 N/S/E/W marks |
| Per-room crop + interior elevation | NativeElementMeshExporter cropRegion; CreateElevationMark room auto-scope | `DS4` computes crop region + 4 interior marks |
| Dimensions / set-out | C34 styles; `produceDimensions` (geometry-kernel) | `DS5` `set-out` mode (offset-from-datum + width + overall) |
| Drawing set / numbering | C30 (`DrawingSetStore`) | `DS6` numbering scheme (§1.4) feeds C30 |
| PDF | C29 (`sheetToPdfBytes`) | `DS2` `sheet.export.pdf` command + book export |

## §3 — CI / verification
- **Purity gate**: the DS1–DS6 modules are covered by the existing schema/kernel purity checks (no
  THREE/DOM/I/O/RNG). Unit tests assert determinism (same input → identical output) for every DS.
- **Coverage gate**: a generated set's `§DIAG`/coverage report MUST list every level + room and flag
  skips (§1.3). A test asserts plan/elevation/set-out/room sheet counts vs the input.
- **P6 gate**: the `docSheets` executor performs no direct store writes (the existing
  `check-direct-store-writes` gate, extended to the executor).
- **Determinism**: `planDocumentationSet` golden test (a fixed building → a fixed numbered plan).
