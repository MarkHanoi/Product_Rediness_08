## §A — Decision

PRYZM 2 self-host ships as a Docker Compose stack with **six** services:

| Service       | Image                                  | Internal port | Exposed |
| ------------- | -------------------------------------- | ------------- | ------- |
| `postgres`    | `postgres:16-alpine`                   | 5432          | —       |
| `minio`       | `minio/minio:RELEASE.2026-01-01T…`     | 9000 / 9001   | 9001    |
| `sync-server` | `ghcr.io/pryzm/sync-server:2.0.0`      | 4000          | —       |
| `bake-worker` | `ghcr.io/pryzm/bake-worker:2.0.0`      | 4001          | —       |
| `api-gateway` | `ghcr.io/pryzm/api-gateway:2.0.0`      | 5101          | —       |
| `editor`      | `ghcr.io/pryzm/editor:2.0.0`           | 80            | **3000**|

The user-facing front door is `http://localhost:3000` (the `editor` nginx),
which:
- serves the editor SPA from the baked-in `/usr/share/nginx/html`,
- reverse-proxies `/v1/*` to `api-gateway:5101`,
- reverse-proxies WebSocket upgrades on `/ws*` to `api-gateway:5101`.

Secrets (Postgres + MinIO root passwords) are 24-byte hex generated by
`install.sh` via `openssl rand -hex 24` and mounted as Docker secrets.
Volumes (`pryzm-postgres-data`, `pryzm-minio-data`) persist across restarts.

`init-db/01-bootstrap.sql` and `init-db/02-marketplace.sql` are mounted into
the postgres container at `/docker-entrypoint-initdb.d/` and run once on first
boot to create `schema_migrations` + the marketplace tables (mirroring
`apps/marketplace-api/migrations/0001_marketplace_plugins.sql`).

## §B — Why six services and not five (spec reconciliation)

The phase-doc YAML at PHASE-3D §S67 lists **five** services and notes that
"editor (served by api-gateway)" is bundled into the api-gateway image.  Two
implementation paths satisfy that intent:

1. **Bundle path** — multi-stage build that compiles the editor SPA and
   copies `dist/` into the api-gateway image at `/app/static`, then teaches
   the api-gateway Express app to serve static files via a new
   `app.use(express.static(staticDir))` middleware gated on a `STATIC_DIR`
   env var.  Five services.

2. **Reverse-proxy path** — keep api-gateway code untouched and add a sixth
   `editor` service running `nginx:1.27-alpine` that serves the SPA and
   reverse-proxies API + WS traffic to api-gateway.  Six services.

We chose **path 2** for three reasons:

1. **Code stability invariant.** The api-gateway has 175 passing tests
   exercising its Express app contract (S65 audit + S66 webhooks composition).
   Adding `express.static` + a new env var would touch `app.ts`, requiring
   re-validation of the auth shim, rate-limit registry, and route ordering.
   Self-host scope is "package what we have," not "modify what we have."

2. **Separation of concerns.** Static asset caching, gzip/brotli, TLS
   termination (post-S70), and CSP header injection (S68 D3) all belong at
   the front door, not in the API server.  nginx is the right tool for that
   layer; running it as its own container cleanly carves it off.

3. **Reverse-proxy is also where TLS lands.** S70 D8 publishes the self-host
   stack with optional TLS — the editor container is the natural place to
   add `ssl_certificate` directives without touching api-gateway.  Having
   it carved off from S67 means S70 is a config-only change.

The cost is one extra container (~30 MB nginx image) and one extra hop on
every API request (negligible at LAN latencies).

## §C — Build context = repository root

Every service Dockerfile uses `context: ..` (the repository root, parent of
`pryzm-selfhost/`) so pnpm can resolve `workspace:*` references during
install.  Without this, `pnpm install --filter @pryzm/api-gateway...` cannot
find the workspace packages it depends on.

`.dockerignore` at the repo root keeps the build context lean by excluding
`node_modules`, `dist`, `attached_assets`, etc.

## §D — Local build vs. ghcr.io publish

The phase-doc compose YAML uses `image: ghcr.io/pryzm/<service>:2.0.0`
without a `build:` directive, implying images are pulled from a public
registry.  As of S67 D1, those images do not exist (ghcr.io publish pipeline
lands at S67 D5 / S70 with CI infrastructure).

**Resolution:** every service entry has both `image:` and `build:` directives.
- `docker compose build` builds locally, tagging with the canonical
  ghcr.io name.
- `docker compose pull` works once images are published (no compose change
  needed on the operator's side).
- `install.sh` runs `build --pull` by default (ensures base images are
  fresh) and supports `SKIP_BUILD=1` for the post-publish path.

## §E — Ports

The phase-doc YAML exposes `api-gateway` on `3000:3000`, but the actual
api-gateway listens on `API_GATEWAY_PORT` (default `5101`).  Per §B, the
front door is `editor` (nginx) on `:80`, mapped to host `:3000`.  api-gateway
runs internally on `:5101` and is not host-exposed.  This satisfies the
phase-doc's "PRYZM live at localhost:3000" exit criterion without colliding
with any other Docker stack on the host that may be using port 5101.

## §F — Healthchecks

Every service has a healthcheck:

| Service       | Probe                                                  |
| ------------- | ------------------------------------------------------ |
| `postgres`    | `pg_isready -U pryzm -d pryzm`                         |
| `minio`       | `mc ready local`                                       |
| `sync-server` | `wget -qO- http://localhost:4000/health`               |
| `bake-worker` | `wget -qO- http://localhost:4001/health`               |
| `api-gateway` | `wget -qO- http://localhost:5101/v1/health`            |
| `editor`      | `wget -qO- http://localhost/healthz`                   |

`install.sh` polls `docker compose ps` and exits with code 0 only when no
service reports `starting` or `unhealthy`.  Default timeout 300s; override
via `HEALTHCHECK_TIMEOUT_SEC=600`.

> **Caveat (honest):** `sync-server` and `bake-worker` healthcheck endpoints
> (`/health`) are referenced in the phase-doc compose verbatim but are not
> yet wired into the service `app.ts` files (sync-server has `app.ts` but no
> `/health` route as of this commit; bake-worker similarly).  These routes
> land at S67 D3 follow-up or at S68 D7 alongside the dependency scan.  Until
> then, the sync-server and bake-worker healthchecks will report
> `unhealthy` after `start_period`, blocking compose from declaring the
> stack ready.  Workaround: run `docker compose up -d` directly (skipping
> `install.sh`'s wait loop), or set `HEALTHCHECK_TIMEOUT_SEC=10` and accept
> the timeout — services still function; they just don't self-report health.

## §H — Consequences

**Positive:**
- Self-host artefacts exist and are committed; an operator with a Docker
  host can `cd pryzm-selfhost && ./install.sh` today and get a buildable
  stack (modulo the healthcheck caveat in §F).
- api-gateway code is untouched (175-test contract preserved).
- Reverse-proxy path means TLS at S70 is a config-only change.
- `init-db/02-marketplace.sql` mirroring of the canonical migration is
  drift-checked in the S68 D7 lint pass.

**Negative:**
- One extra container vs. the spec's five.  Justified by §B.
- Healthchecks for sync-server + bake-worker need follow-up wiring.
- No fresh-VM test was performed in this commit; the D6 verification gate
  is honestly marked deferred, not closed.

**Reversibility:** high.  Switching to the bundle path (§B option 1) is a
single PR: add `express.static` middleware to api-gateway behind an env var,
delete the `editor` service, change the api-gateway port mapping to
`3000:5101`.  The `init-db/`, secrets, install.sh, and Dockerfiles for
sync-server + bake-worker would not need to change.
