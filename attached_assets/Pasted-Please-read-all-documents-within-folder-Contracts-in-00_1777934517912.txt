3. Bottleneck 1: O(n²) accumulation in the command loop — DOMINANT (≈ 2210 ms)
This is the root cause of the NFT breach. The per-slab times grow monotonically as the curtainWallStore grows, producing the characteristic curve above.

Where it comes from:

Inside CreateCurtainWallsOnAllSlabsCommand._processSlabs(), for each slab in the outer loop, the command calls curtainWallStore.add() (or equivalent) once per wall on that slab — creating 17–21 walls per iteration. Each curtainWallStore.add() call runs synchronously and, critically, triggers the store's internal subscriber notification path.

Although storeEventBus.batch() is active (depth ≥ 1), the store-internal subscriber loop runs before the event reaches the bus — this is not buffered by the bus. The store notifies its own listeners (which include _pendingBuildsMap dedup guards, geometry-builder listeners, CurtainPanelSyncHandler, RoomTopologyObserver, and others) synchronously, inline, per add() call.

Because these listeners iterate over Map entries or subscriber arrays that grow with the store size, the per-add() cost is O(current store size). After N slabs have been processed (each with W walls), the store has N×W walls. Processing slab N+1 means each of its W add() calls costs O(N×W). The total inner loop cost across all slabs is:

Σ(i=1 to S) W × O(i × W) = O(S² × W²)

For S=21 slabs, W≈11 walls/slab: 21² × 11² ≈ 53,361 units of work — matches the observed growth.

The doc 34 §9.4 diagnosis confirms this specifically:

"root cause is O(n²) wall-intersection detection as wall count grows per slab. Requires SlabWallCoupling / spatial-query optimization."

The Wall-Intersection detection (checking whether a new curtain wall collides with or couples to existing walls) is the hottest inner operation. RoomTopologyObserver is attached to CurtainWallStore (confirmed in the boot log: [RoomTopologyObserver] attached to CurtainWallStore). Every add() call on the curtain wall store causes the observer to re-evaluate spatial relationships against the full current wall set.

Fix path:

There are two independent interventions:

Fix 3a — addMany() API on curtainWallStore (C11 §4.2 required for batch paths): Instead of calling store.add(wall) N times, call store.addMany(walls) once per slab. The store fires subscriber notifications once (after all items are committed), not once per item. The RoomTopologyObserver and CurtainPanelSyncHandler listeners execute once per slab batch instead of once per wall. This converts the dominant inner cost from O(W_per_slab × current_store_size) to O(1) notification firings per slab, reducing total complexity from O(S²×W²) toward O(S×W).

Fix 3b — Spatial index for wall-intersection queries: The SlabWallCoupling query that currently iterates the full curtain wall store to find intersections should use a spatial index (e.g., topologySpatialIndex — already initialised in the boot log). Replace the O(n) linear scan with an O(log n) range query. This caps the per-wall cost at O(log total_walls) regardless of store size.

Both fixes are required — 3a eliminates the notification-loop accumulation; 3b eliminates the spatial scan accumulation. Either alone gives partial improvement; both together bring the 21-slab case from ~2210 ms toward the sub-500 ms NFT.

4. Bottleneck 2: BimManager serial registration drain — SECONDARY (visible but post-COMPLETE)
The log shows 231 sequential lines of:

[BimManager] Registered element <uuid> to level L-0x-<timestamp>-x

These fire synchronously in one block after signalBuildQueueDrained() (i.e., after the rAF drain empties). Each BimManager.registerElement() call appends to the level's element list and fires a level-update notification. With 231 walls across 11 levels, this is 231 sequential notify-and-update cycles.

This is not in the 2210 ms window — it contributes to the time-to-first-visible-geometry gap (the user sees the loading indicator go away, then watches the building appear incrementally). In the NormalCW log, the registration block takes ~300–400 ms and is the main "after-COMPLETE latency" visible to the user.

This registration drain does NOT use addMany() or batch notification. BimManager receives individual curtainwall-added events from endBatchYielded() chunks, and processes each one synchronously when it arrives.

Fix path:

Introduce a BimManager.registerMany(elements[]) method that does one pass over the element array, groups by level, and fires one notification per level (not one per element). Call this from the endBatchYielded() onComplete callback (or from a dedicated onBatchComplete hook in BatchCoordinator). For 231 walls across 11 levels, this replaces 231 notify calls with 11 — a 21× reduction.

Alternatively, the existing endBatchYielded() chunked dispatch already groups events by frame (200 per 'pre-render' frame). If BimManager subscribes to the bus and processes the entire chunk at once rather than registering a per-element listener, the registration could be batch-grouped at the chunk boundary. This requires no new API — only a change to how BimManager subscribes.
