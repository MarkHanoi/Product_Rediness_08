## BN-01 — `addMany()` internal subscriber cascade (978 ms here; 9,706 ms at 273 walls)

**What the log says:**
```
PHASE-2-ADDMANY-DONE count=147 addManyMs=978.5ms busDepthNow=2
LONGTASK duration=1026.0ms
```

**Root cause (confirmed, not just theoretical):** `storeEventBus` is at depth=2 during `addMany()`, so bus-level events are buffered. But the `CurtainWallStore`'s internal per-item write path still fires synchronously on every insert. The 42-BOTTLENECK doc correctly identified this. Each insert hits:

1. `RoomTopologyObserver._scheduleDebounce()` — the `isBatching` guard in the code you shared (`storeEventBus.isBatching()`) only checks the bus depth. At depth=2, `isBatching()` returns `true` — so the guard **should** be firing. But the log from the 273-wall session shows `reset (level=L0, n=1..6)` happening anyway. This means the RTO is being reached via a path that bypasses the `isBatching()` check — most likely a direct `WallStore` listener rather than the curtain wall store, or the guard was added to `RoomTopologyObserver` but not to the listener wired to `CurtainWallStore`.

2. `TopologySpatialIndex` rebuild — rebuilds on every `Map.set()` call in the store.

3. Any `ConstraintEngine` subscriber wired to the store's internal change hook.

The scaling exponent (n^4.77 at 273 walls) means this blows up badly at 200+ walls even if 147 walls is "only" ~1 second.

**Fix (FX-01) — exact implementation needed:**

`CurtainWallStore` needs a `_batchMode` flag that is set before the `addMany()` loop and cleared after. Every internal subscriber (topology observer, spatial index, constraint engine) must check this flag and defer work until it clears:

```typescript
// CurtainWallStore.addMany()
addMany(walls: CurtainWallData[]): void {
  this._batchMode = true;
  try {
    for (const wall of walls) {
      this._walls.set(wall.id, wall);  // raw Map write — no internal notifications
    }
    // ONE consolidated notification after all inserts
    storeEventBus.emit({ type: 'curtainwall', action: 'add-many', ids: walls.map(w => w.id) });
  } finally {
    this._batchMode = false;
    this._onBatchModeEnd();  // signal deferred subscribers to run once
  }
}
```

The `RoomTopologyObserver` subscriber wired to `CurtainWallStore` must check `store.isBatchMode()` (or `batchCoordinator.isBatching`) before resetting its debounce timer — and the check must be on the correct curtain wall store path, not just the wall store path.

**Expected reduction: 978 ms → ~30 ms for 147 walls. At 273 walls: 9,706 ms → ~60 ms.**

## BN-02 — First-render PSO compile (13,303 ms LONGTASK after suppress lifts)

**What the log says:**
```
FIRST-RENDER-POST-SUPPRESS totalSuppressedMs=1202.9ms frameCount=1019
LONGTASK duration=13,303.0ms start=434067.8ms
ON-BATCH-END-DONE T=+14503.3ms
```

**Why this is happening even though prewarm ran:** The log shows `shadersPrewarmed=true` at command start — the prewarm ran during a *previous* batch in the same session (the 20-slab run). But the first render LONGTASK is 13,303 ms. The 42-BOTTLENECK doc §11.2 explains this: the prewarm covered the 3 basic PSO variants (mullion IM, glass IM, fallback Mesh), but 147 additional `InstancedMesh` objects in the scene cause the WebGPU driver to compile additional PSO variants on the first production render — specifically the combination of the SSGI denoiser reading a GBuffer that now contains 147 IM depth values, plus the outline pass edge-detecting across a scene that doubled in IM count.

The 3-pass prewarm (`rpm.render(0)` × 3) was meant to cover SSGI+TRAA PSOs. But the PSO key in WebGPU includes the **count of active render targets and the specific geometry buffer layouts currently bound**. Adding 147 new `InstancedMesh` objects changes the scene's render target layout enough to invalidate the cached PSO for those pipeline phases.

**Fix (FX-02) — two options, pick one:**

*Option A (architectural, recommended):* After the rAF drain completes and geometry is in the scene, but before lifting render suppression, call `rpm.render(0)` once more with the real geometry present. This warms the PSOs in the exact production scene state. Timing: inside `signalBuildQueueDrained()` → before `endBatchRenderSuppress()`.

```typescript
// BatchCoordinator._drainRegistrations(), after registration drain:
// Call one warm render with real geometry before unblocking:
try { window.renderPipelineManager?.render(0); } catch { /* non-fatal */ }
// THEN lift suppression:
viewDependencyTracker.setSuppressed(false);
unifiedFrameLoop.endBatchRenderSuppress();
```

This adds ~200–500 ms hidden under the overlay but eliminates the 13,303 ms user-visible LONGTASK. Net gain: ~12.8 s.

*Option B (quick fix):* Increase the prewarm to render 5–6 passes instead of 3, to better saturate the SSGI denoiser and TRAA accumulation buffer before the real geometry arrives. Less reliable because it doesn't use the actual scene geometry.
