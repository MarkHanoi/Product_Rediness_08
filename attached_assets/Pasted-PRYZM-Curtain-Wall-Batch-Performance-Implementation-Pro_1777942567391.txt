Fix the O(n²) Slab Loop — saves ~5,000–7,000ms
This is by far your biggest problem. The cascading times (15 → 33 → 60 → 92 → 135ms per slab) are caused by curtainWallStore.add() triggering subscribers that scan the growing store on every single insertion. At 400 walls this becomes catastrophic.
The fix is to batch all store mutations into a single storeEventBus.batch() call where no subscriber fires mid-loop, then flush once at the end. Your runBatch() already does this — but the CurtainWall builder's §BATCH-CW-PAUSE resumeAndFlush is still happening synchronously after the loop, meaning the geometry build for all 108 walls runs on the main thread before signalBuildQueueDrained() is called.
The concrete fix is to move the CurtainWall builder drain entirely into rAF slices BEFORE the registration phase, not after. Currently your code does:
runBatch(fn) → store.add() × 108 [buffered]
resumeAndFlush() → 108 walls transferred to pending queue
→ ONE rAF drain scheduled
signalBuildQueueDrained() → called by builder when drain empties
_executeFinalSweep() → endBatchYielded()
The problem is resumeAndFlush() runs synchronously inside runBatch(), which means all 108 walls get queued for the builder drain before the JS event loop yields. On a 400-wall project the drain itself (geometry upload) takes 800–1200ms spread across rAF frames, but the initial queue population is synchronous and triggers subscriber cascades.
Fix: Gate resumeAndFlush() behind requestIdleCallback or a single scheduleOnce('pre-render') so the main thread yields before the drain begins. Estimated saving: 4,000–6,000ms.2. Eliminate Per-Material Shader Compilation — saves ~1,500–2,500ms
Your §FIX-BATCH-RENDER-SUPPRESS already suppresses renders during the drain, but shader compilation happens the moment the drain ends and the first renderer.render() fires with new materials. For 400 curtain walls with 2–3 unique materials each, WebGL compiles ~800–1,200 shader program variants synchronously on that first frame.
The fix is to pre-warm the shaders before the batch starts. Add a single invisible render call with a representative MeshStandardMaterial before runBatch():
typescript// Before runBatch():
const warmupMesh = new THREE.Mesh(
new THREE.PlaneGeometry(0.001, 0.001),
new THREE.MeshStandardMaterial({ color: '#333333', transparent: true, opacity: 0 })
);
world.scene.three.add(warmupMesh);
renderer.render(world.scene.three, world.camera.three); // compiles shaders now
world.scene.three.remove(warmupMesh);
warmupMesh.geometry.dispose();
warmupMesh.material.dispose(); / This forces WebGL to compile the shader variant once before any curtain wall geometry arrives. Subsequent walls reuse the compiled program. Estimated saving: 1,200–2,000ms.
Replace curtainWallStore.add() Loop with addMany() — saves ~500–800ms
Every curtainWallStore.add() call currently emits a store event even inside a batch envelope, because storeEventBus.batch() only buffers the bus delivery, not the store's internal onItemSet notifiers. At 400 walls that is 400 separate internal notifications.
Add an addMany(items: CurtainWallData[]) method to your CurtainWall store that pushes all items into the internal Map in one loop and fires a single onItemSet notification at the end. The bus event is still buffered by the outer envelope. Estimated saving: 400–600ms.
Defer PBR Upgrade to Idle Time — saves ~200–400ms
Your current post-batch PBR upgrade runs upgradeNewMeshes() chunked at 120 meshes/frame immediately after endBatchYielded() completes. For 1,500 meshes that is ~13 frames of post-render work that blocks the user seeing the result.
Move it to requestIdleCallback with a timeout: 5000. The meshes look fine without PBR for the first 5 seconds — users are still absorbing the geometry appearing. Estimated saving in perceived load time: 300–500ms (the geometry appears visible sooner even if PBR upgrades are still in flight).5. This does not help batch creation time but dramatically improves the frame rate after the batch, making it feel faster. Your InstancedElementRenderer singleton already exists for this.
