The three specific fixes
Fix R1 — Prewarm must cover the actual instance counts in the batch, not just 3 arbitrary probes.
The current prewarm creates one InstancedMesh with count=1 for each variant. WebGPU PSO descriptors include the indirect buffer layout, which varies with count. Before the batch starts you know exactly how many walls and what geometry they'll produce. Build prewarm probes with representative instance counts — specifically the largest count you'll actually render. One probe with count = maxMullionsPerWall, one with count = maxPanelsPerWall. That covers the worst-case PSO descriptor and the GPU driver will promote lower counts as cache hits.
Alternatively, after runBatch() returns but before suppression lifts, call rpm.render(0) once more with the actual scene geometry in place (during the rAF drain, the geometry is already in the scene but suppress is still on — you could fire one render pass at T+BUILD_DRAIN_COMPLETE to warm PSOs against the real objects).
Fix R2 — EdgeProjector must not fire on the first post-suppress frame.
The 300ms debounce in markLevelsDirty is not enough. The PSO LONGTASK alone is 5,293ms, which means the debounce timer has long expired by the time the first frame renders, and the queueLowPriority task is picked up on that exact first frame. You need to defer the plan-view projection until after the scene is stable.
The clean fix is to add a postSuppressRenderCount guard in UnifiedFrameLoop. After suppression lifts, count rendered frames. Only call markLevelsDirty after N stable frames (N=3 is enough — typically 50–100ms at 60fps). This keeps the PSO compile LONGTASK isolated and allows the GPU to fully drain its command queue before EdgeProjector adds its own raycasting load.

Fix R3 — Shadow reactivation must not use requestIdleCallback during a LONGTASK cascade.
requestIdleCallback is designed for when the browser is genuinely idle. During a 20-second LONGTASK storm it either never fires (starved) or fires between LONGTASKs at the worst possible moment. The current setTimeout(10000) for the first slice was calibrated for a 9-second storm but the storm runs 21 seconds.
Replace with a proper post-stable callback: wire _reactivateShadows() to fire via the same _onPostSuppressStable hook from Fix R2, but deferred further — after postSuppressStabilizeFrames + 5 more frames (roughly 80ms after the scene is fully stable). This guarantees shadows reactivate into a quiet main thread, not into an active LONGTASK. Remove the requestIdleCallback calls from drainSlice too — replace with setTimeout(drainSlice, 200) as a minimum spacing between slices (200ms is longer than any single EdgeProjector chunk and allows the GPU to breathe).

Why the "DONE" message fires at 5 seconds but the scene isn't live for 47 seconds
The COMPLETE walls=66 totalMs=292.9ms and the overlay dismissal both happen at the end of ON-BATCH-END-DONE (~T+897ms from batch start). At that point the geometry is registered, the overlay is gone, and suppression lifted is logged. The code considers the batch done. But the main thread is then immediately occupied for another 45 seconds by R1+R2+R3. The batch pipeline's accounting ends at "geometry in scene + events delivered" — it doesn't account for the PSO compile cost, EdgeProjector cost, or shadow rebuild cost that follow. Those are considered "first-render side effects," but they're actually serial dependencies that the user experiences as the batch not being complete.
The fix for the expectation mismatch is to keep a lightweight "scene settling" indicator (not the full batch overlay) visible until _onPostSuppressStable fires — that's the real moment the scene becomes interactive.
