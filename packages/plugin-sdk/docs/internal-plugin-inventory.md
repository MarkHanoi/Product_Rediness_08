## Migration status legend

- **`built-in-no-migration-needed`** — the plugin is statically linked
  into the editor bundle via `apps/editor/src/PluginRegistry.ts`'s
  `ALL_PLUGINS` array. It uses the internal `PluginDescriptor` interface
  in `PluginRegistry.ts`, NOT the public `@pryzm/plugin-sdk`
  `PluginManifest` schema. SDK 1.0 publish does not require any change
  to these plugins; they continue to use the internal contract. (See
  ADR-0038 §Decision C: the internal type stays named `PluginDescriptor`
  through S62 D1; D3 host-proxy sprint will rename it to
  `InternalPluginRecord` to free the public name.)
- **`already-uses-descriptor-pattern`** — the plugin has its own
  ad-hoc `descriptor.ts` file (the five `plugins/ai-*` packages). These
  will be the first to migrate to the public SDK at D5 (example
  plugins) because they already follow the descriptor-as-data pattern.
- **`needs-manifest`** — the plugin has no descriptor file and is
  consumed via direct imports. Migrating it to the public SDK
  requires (a) adding a `plugin.manifest.json` and (b) wiring it
  through the iframe sandbox. Scheduled for post-S62 (S63 docs site
  uses these as examples).

## What this inventory does NOT cover (scope-limited at D1)

- Per-plugin manifest *content* — the `needs-manifest` rows are
  enumerated, but the actual `plugin.manifest.json` files for them are
  D2-D5 work (after host proxies + sandbox + example plugins land).
- Marketplace-side metadata (icons, screenshots, pricing) — that is S64
  marketplace work.
- Signing keys per plugin — that is D8 signing-infra work.
- Hot-reload `pryzm dev` per-plugin support — that is the D4 sandbox
  + D6 SDK README path.

```bash
  | sort \
  | xargs -I{} bash -c 'echo "$(jq -r .name {}) | {}"'
```

the aggregate counts must be re-tallied before the D9 K3-C gate is
